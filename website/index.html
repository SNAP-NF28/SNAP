Présentation

Overview

SNAP est un projet réalisé dans le cadre de l’UV NF28 à l’Université de Technologie de Compiègne.

Afin d’étudier la conception et le développement de systèmes interactifs, nous proposons de décliner un site Web applicatif sous la forme de plusieurs interfaces, adaptées à différents types de terminaux. Nous avons ainsi tiré parti des dernières technologies Web afin de proposer une personnalisation avancée de l’expérience utilisateur. En particulier, nous proposons des interfaces dédiées pour les ordinateurs, les smartphones et les tablettes.

Notre site permet à l’utilisateur de faire l’agrégation des fils d’actualités de ses différents réseaux sociaux. En plus de l’agrégation des messages, SNAP permet aussi à l’utilisateur de poster des messages sur un ou plusieurs réseaux sociaux simultanément parmi ceux actuellement intégrés dans le site, soit Facebook et Twitter.

Nous avons accordé une attention particulière à l’étude des interactions de l’utilisateur ainsi que des contextes d'utilisation associés aux différents périphériques. En effet, nous cherchons une interface la plus adaptée au terminal utilisé, et donc aux interactions propres à ce terminal.


Tester le site, récupérer le code ou les documents

SNAP is a project which has been realized (achieved) within the NF28 course, in the University of Technology of Compiegne.

In order to study the design and development of interactive systems, we developed a web application with several interfaces, all adjusted for different devices. The latest web technologies has been exploited in order to offer a custom experience to the user. This goal is achieved by providing dedicated interfaces for computers, smartphones and tablets.

This website allows the user to merge the newsfeeds of different social networks. Furthermore, SNAP application enables the user to post messages on several social networks simultaneously among the available ones, ie Facebook or Twitter.

Special care has been taken to study and understand user interactions, as much as the different environments related to these different devices (computer, smartphone or tablet). Indeed, the user interface has to be adapted to the device and its characteristics.

Aspects techniques

Technical aspects

Côté serveur

Notre site web comporte une partie serveur réduite, et nous avons favorisé l'exécution de code côté client. En effet, la tâche du serveur est de fournir les scripts spécifiques au périphérique utilisé, aucun traitement n'est réalisé à distance.

Nous avons tout de même déployé notre site sur Heroku, un serveur distant, en développant un serveur basé sur Node.js, et ce pour plusieurs raisons :

Prise en charge de différents périphériques
Les terminaux autres que les ordinateurs ne gèrent pas les interfaces logiques de type localhost
Dépendance face à la configuration de la machine
La nécessité d'un déploiement extérieur se fait aussi sentir dans le cadre la compatibilité du site sur plusieurs navigateurs et pour des configurations différentes
Temps de chargement
Les tests en local font que les scripts sont chargés quasi-instantanément, ce qui ne correspond pas au cheminement réel des données sur le Web

Côté client

Amélioration du code


Nous avons utilisé le framework JavaScript Angular.js pour pouvoir structurer notre code. Ce framework, originellement créé pour gérer du contenu exclusivement statique, permet de s'affranchir de la gestion des appels asynchrones. En effet, il implémente un ensemble de mécanismes pour faciliter la gestion du code via le design pattern Modèle-Vue-Contrôleur, notamment le two-way binding (lien à double sens, donc entre la vue et le modèle). Le code produit, fortement influencé par le modèle MVC, est modulaire.

Toutefois, l'utilisation de ce framework n'est pas dénuée de défauts : Angular.js est une technologie très récente, son utilisation ne s'est pas encore démocratisée, d'où un certain manque de documentation. De plus, de nombreuses actions sont réalisées automatiquement par ce framework, d'où parfois une "mystification" du fonctionnement du site.

Ergonomie & esthétique


L'objectif du projet étant de tirer parti des dernières avancées technologies pour réaliser une interface ergonomique et personnalisée, nous avons donc utilisé les propriétés offertes par HTML5 et CSS3, notamment les APIs de sessionStorage et le manifest (pour HTML5), et certaines propriétés CSS3 telles que les coins arrondis (border-radius), les ombres (box-shadow, text-shadow) ou les transitions (transition).

JQuery Mobile est un framework graphique libre porté par communauté JQuery, et compatible avec la plupart des navigateurs mobiles du marché. Nous l'avons donc utilisé pour la réalisation des interfaces smartphone et tablette. De plus, le rendu visuel des composants se rapproche de celui des applications natives des systèmes d'exploitation mobiles (i.e. iOS et android).

Cependant, un des points critiques de ce framework est son manque d’interopérabilité avec Angular.js. En effet, ces deux frameworks procèdent à des modifications automatiques du Document Object Model ("mystification"). Nous avons donc trouvé un adaptateur JQuery Mobile/Angular open-source fourni par un développeur indépendant pour remédier à ce problème.

Bootstrap est un framework graphique développé par et pour twitter, offert ensuite au monde du libre. Nous avons l'utilisé principalement pour l’interface pour ordinateur, où toute la partie graphique est prise en charge par Bootstrap, et pour l’écran de chargement et de détection du terminal.

Utilitaires


L'utilisation de la bibliothèque JavaScript JQuery est très répandue, en fait très rares sont les applications JavaScript qui n'en font pas usage. JQuery fournit de nombreux outils pour faciliter la plupart des opérations communes et fastidieuses : manipulation du DOM, implémentation de systèmes AJAX.... Son utilisation s'est donc imposée dans le cadre de notre projet.

Formfactor est une petite bibliothèque JavaScript encore au stade expérimental, et permet de tester certaines caractéristiques du navigateur web (principalement à l’aide du DOM et de bibliothèques tierces comme modernizr.js), et de moduler le comportement de la page en fonction du résultat de ces tests. Dans le cadre de notre projet, nous avons donc pu ainsi tester la résolution de l'écran, le user-agent, et la présence de fonctionnalités telles que le tactile, pour récupérer le type de terminal sur lequel l'utilisateur se trouve, et donc rediriger ce dernier vers l'interface la plus adaptée (tablette, smartphone ou desktop).

RequireJS est une bibliothèque JavaScript généralement utilisée pour le chargement asynchrone de scripts, même si le nombre de fonctionnalités offertes par cette bibliothèque a beaucoup augmenté depuis ses premières versions. Le chargement de script représente un point primordial dans notre projet, dans la mesure où nous avons développé de nombreux scripts, parfois lourds, alors qu'un des points critiques de notre site est sa réactivité.

L'utilisation des Software Development Kits (SDK) développés par les réseaux sociaux ne s'est pas faite sans mal. Tout d'abord, les bibliothèques fournies par les réseaux sociaux sont beaucoup plus fréquentes pour les langages serveurs que pour les langages clients. Celà est dû au fait que les développeurs concentrent en général leur travail côté serveur plutôt que côté client, mais c'est aussi dû au mode d'authentification sur les réseaux sociaux. En effet, tout le code exécuté côté client est directement accessible et visible en clair par ce même client. Le développement de SDK sécurisés côté client requiert un surcoût en travail pour les réseaux sociaux, qui sont donc plus réticents à développer ce type de solution.

Nous avons donc utilisé les frameworks directement fournis par Facebook et twitter, celui fourni par Facebook étant mieux documenté que celui de twitter. Nous n’avons pas pu nous procurer un SDK JavaScript pour Google+. C’est d’ailleurs la raison pour laquelle nous n’avons pas intégré ce réseau social dans notre projet.

Server side

Our website is made of a tiny server side, we have favored client-side code. Indeed, no processing is done on the server, its task is only to provide the right scripts when asked to.

We still deployed our website on Heroku, a cloud application platform, with a server based on Node.js. We did it because of several reasons :

Support of different devices
Non-computer devices (in our case, tablets and smartphones) do not support loopback network interfaces like localhost
Dependencies on the device’s configuration
In order to maximize our website’s compatibility with as much browsers and devices as possible, we had to deploy our website on an external server
Loading time
In case of local tests, scripts are almost immediately loaded, which does not reflect the reality of packages travelling on the Web

Client side

Code improvement

We used the JavaScript framework Angular.js to structure our code. This framework, originally created to manage static content, frees us from managing asynchronous calls. It indeed implements functionalities for easier code managing through the Model-View-Controller design pattern, and more specifically with two-way binding between the view and the model. The produced code is very modular.

However, this framework can be bothersome: Angular.js is a recent technology; its use is not widespread yet, so it lacks some documentation. Moreover, many actions are done automatically by this framework, so there are sometimes “magical things” happening.

Ergonomics & aesthetics

To achieve the main goal of the projet, which was to benefit from the latest Web technologies in order to develop ergonomic and customized interfaces, we used HTML5 and CSS3 properties : sessionStorage APIs and the manifest for HTML5, and some CSS3 properties like rounded corners(border-radius), shadows (box-shadow, text-shadow) and transitions (transition).

JQuery Mobile is a graphic open-source framework, maintained by the JQuery community, and compatible with most of the commonly used mobile Web browsers. We used it to render the tablet and smartphone interfaces. Furthermore, the look of its components is very similar to that of native mobile applications (e.g iOS or android).

Nevertheless, one of its drawbacks is its lack of compatibily with Angular.js: These 2 frameworks proceed to automatic Document Object Model modifications (“magical things”). We found an open-source JQuery Mobile/Angular adapter to solve this compatibility problem.

Bootstrap is a graphic framework developed by and for twitter, which has then been given to the open-source community. We mainly used it for the desktop interface, where all the graphics are managed by Bootstrap, and for the loading screen.

Utility libraries

The use of the JavaScript JQuery library is widespread. In fact, only few JavaScript applications do not use it. JQuery provides many tools to solve easily the most common and tedious operations: DOM manipulation, implementation of AJAX systems .... Its use has thus been an obvious choice for our project.

Formfactor is a small experimental JavaScript library, is used to test features of the web browser (mainly using the DOM and third-party libraries such as modernizr.js), and modulate the behavior of the page depending on the result of these tests. During our project, the resolution of the screen, the user-agent and the presence of features such as touch have been tested in order to retrieve the type of device being used, and then redirect the user towards the most appropriate interface (tablet, smartphone or desktop).

RequireJS is a JavaScript library is typically used for asynchronous loading of scripts, although the number of features offered has grown considerably since its early versions. The loading script is a paramount point to our project, as many scripts, sometimes large, have been developed, whereas one of the most critical point of the site is its reactivity.

The use of Software Development Kits (SDK) developed by social networks was not without its difficulties. First, the libraries provided by the social networks are much more common for server languages than for client-side languages. This is because developers typically focus their work on the server side rather than on the client side, but it is also due to the authentication mode on social networks. Indeed, all the code executed on the client side is directly accessible and apparent by the client. The development of secure client-side SDK requires an additional cost of labor for social networks, which are therefore more reluctant to develop this type of solution.

We therefore used the frameworks provided directly by Facebook and twitter, the one provided by Facebook being better documented than twitter. We have not been able to get a JavaScript SDK for Google+. This is why we have not included this social network in our project.
